View 1 — Mapa do Parque Gerador

Fonte: raw_siga_usina + dim_usina_siga · ~10.000+ empreendimentos georreferenciados

Esta é a visão de maior riqueza geográfica do projeto. Cada usina tem latitude/longitude e pode ser plotada diretamente.

🗺 Mapa de pontos

Todas as usinas — por fase e tipo

  • Ponto por CEG (usina)
  • Cor: Sol Eól Hidro Term
  • Opacidade/forma: fase (Operação / Construção / Planejamento)
  • Raio: proporcional a MW (outorgada ou fiscalizada)
  • Tooltip: CEG, nome, UF, MW, data entrada op.
Revela concentração geográfica de cada fonte, fronteiras de subsistema e "buracos" de renovável.
📊 Sunburst / Treemap

Hierarquia do parque

  • Nível 1: Subsistema (N, NE, S, SE_CO)
  • Nível 2: Família (renovável, hidro, térmica)
  • Nível 3: Tipo SIGA (UFV, EOL, UHE, PCH, UTE…)
  • Área: MW fiscalizada
  • Separável por fase
Mostra peso relativo de cada tecnologia dentro de cada região — base para pensar onde há espaço para expansão.
📉 Histograma

Distribuição de portes (MW)

  • Histograma log-scale de MW por usina
  • Separado por tecnologia
  • Mostra bimodalidade (PCH vs. UHE, mini-solar vs. grande)
Relevante para calibrar o bloco_mw mínimo de expansão — se há muitas usinas pequenas, blocos menores fazem sentido.
📅 Timeline

Histórico de entrada em operação

  • Eixo X: DatEntradaOperacao (ano)
  • Eixo Y: MW acumulado ou por safra
  • Stacked por tecnologia
  • Linhas verticais: marcos (Proinfa, leilões de renováveis)
Revela vida útil remanescente do parque — usinas antigas têm maior probabilidade de descomissionamento no horizonte do plano.

View 2 — Curvas de Demanda

Fonte: raw_carga_energia + stg_carga_diaria_mwmed · Diário × 4 subsistemas · ~1.460 pontos/ano

A demanda é capturada em resolução diária — a granularidade máxima disponível. O mart perde a variação intra-mensal ao calcular a média mensal.

📈 Série temporal

Carga diária 2025 — todos os subsistemas

  • Eixo X: data (1 Jan → 31 Dez 2025)
  • 4 linhas: N, NE, S, SE_CO
  • Banda de confiança: ±1σ mensal
  • Marcadores: média mensal (o que o mart usa)
  • Hover: valor exato, dia da semana
Mostra quanto a hipótese de demanda mensal constante do mart é uma simplificação — picos de verão/inverno são visíveis.
🗓 Heatmap calendário

Intensidade de carga por dia do ano

  • GitHub-style: semanas × meses
  • Cor: MWmed (azul=baixo → vermelho=alto)
  • Um heatmap por subsistema (4 painéis)
  • Revela sazonalidade, feriados, fenômenos climáticos
Subsistemas têm sazonalidades distintas — SE_CO (ar condicionado verão) vs. S (aquecimento inverno).
📊 Box-plot mensal

Dispersão da carga por mês

  • Eixo X: mês (Jan–Dez)
  • Box: IQR dos dias do mês
  • Bigodes: min/max
  • Ponto central: média mensal (usado no mart)
  • Facetado por subsistema
Quantifica o erro introduzido pela média mensal — meses com alta variância (dezembro, julho) têm maior risco de subdimensionamento.
🔵 Scatter correlação

Correlação entre subsistemas

  • Matriz de scatter: NE × SE_CO, N × NE, S × SE_CO
  • Ponto por dia
  • Cor: mês
  • Linha de regressão
  • R² exibido
Alta correlação entre subsistemas reduz o benefício do intercâmbio — demandas simultâneas não se compensam.

View 3 — Perfis de Disponibilidade Renovável

Fonte: raw_fator_capacidade + stg_fc_horario_ponderado · Horário × usina · ~8.760h

Esta é a camada de maior granularidade temporal do projeto. Os dados cobrem cada hora de 2025 para cada usina eólica e solar, separadas por subsistema.

🔥 Heatmap hora×mês

Fator de capacidade médio — hora do dia × mês

  • Eixo X: mês (1–12)
  • Eixo Y: hora (0–23)
  • Cor: FC médio (0.0 → 1.0)
  • 8 painéis: 4 subsistemas × 2 techs (sol, eol)
  • Este é exatamente o dado que vira mart_disponibilidade
Solar tem padrão cônico claro (pico 10h–15h, zero à noite). Eólico é mais noturno/madrugada em algumas regiões — complementaridade operacional direta.
📈 Série horária raw

FC horário ponderado — série completa 2025

  • 8.760 pontos por (subsistema, tech)
  • Linha: FC ponderado pela capacidade instalada
  • Banda: min–max entre usinas do subsistema
  • Zoom interativo por período
A banda min–max revela diversidade intra-subsistema — se as usinas são muito heterogêneas, o FC médio ponderado pode mascarar desequilíbrios geográficos.
🕐 Curva de duração

Curva de duração do FC (CDG)

  • FC ordenado de maior a menor
  • Eixo X: % das horas do ano
  • Eixo Y: FC
  • Uma curva por (subsistema, tech)
  • Área hachurada: FC > 0.20 (limiar típico de viabilidade)
Quantifica o complemento de armazenamento necessário: área abaixo de ~0.05 FC representa horas de "calma" que precisariam de backup ou carga flexível.
🌐 Correlação cruzada subsistemas

Correlação eólica entre subsistemas

  • Scatter: FC_NE × FC_S (vento)
  • Ponto por hora
  • Cor: mês
  • Baixa correlação = potencial de intercâmbio
  • Alta correlação = risco sistêmico simultâneo
Se NE e S têm eólico descorrelacionado, o intercâmbio N↔S ampliaria o potencial renovável sem aumentar o risco de apagão.

View 4 — Rede de Transmissão

Fonte: raw_siget_linha + stg_siget_interligacao + mart_arco · Linhas individuais → pares aggregados → arcos de modelo

A transmissão tem 3 níveis de granularidade — visualizá-los juntos mostra exatamente onde o modelo simplifica a realidade.

📊 Barras empilhadas

Capacidade por par e nível de tensão

  • Eixo X: 6 pares inter-regionais (N↔NE, NE↔SE_CO, SE_CO↔S, N↔SE_CO, NE↔S, N↔S)
  • Barras empilhadas: 230 / 345 / 440 / 500 / 525 / 800 kV
  • Valor: MVA total por nível
  • Marcador: capacidade final usada no mart_arco
O mart soma todos os níveis de tensão em um único número — esta viz mostra se um par depende de muitas linhas antigas de 230kV ou de poucas linhas modernas de 800kV.
🕸 Chord diagram / Matriz

Matriz de capacidade inter-subsistema

  • Matriz 4×4: N, NE, S, SE_CO
  • Célula: MVA total disponível
  • Cor: utilização relativa (cap. / demanda destino)
  • Diagonal: zero (sem loop)
Identifica assimetrias: N→SE_CO pode ter capacidade muito diferente de SE_CO→N dependendo do direcionamento dos contratos.
📊 Scatter km × MVA

Eficiência das linhas (MVA por km)

  • Eixo X: comprimento km (NumEtnLinTms_km)
  • Eixo Y: MVA estimado
  • Cor: tensão (kV)
  • Tamanho: circuitos
  • Filtro: apenas ativas
Linhas longas de baixa tensão têm baixa eficiência — candidatas a reforço ou substituição por HVDC em horizontes de planejamento mais longos.
📈 Histograma

Distribuição de comprimentos e tensões

  • Histograma de NumEtnLinTms_km
  • Stacked por nível de tensão
  • Separado por status (Ativa / Desativada)
  • Km total e n_circuitos como KPIs

View 5 — Parâmetros Econômicos

Fonte: raw_pde_parametro + mart_parametro_tech · Tecnologias × variantes de custo

Apesar de pequena em volume, esta tabela concentra os pressupostos mais impactantes para o resultado do modelo. Visualizá-la com sensibilidade é essencial.

📊 Grouped bar

CAPEX, OM e CVU por tecnologia

  • 4 grupos: sol, eol, hidro, term
  • 3 barras por grupo: CAPEX (R$/kW), OM fixo (R$/kW/ano), CVU (R$/MWh)
  • Eixos duplos (escala diferente para CVU)
  • Referência: câmbio BRL/USD Dec 2024 = 6,10
📈 Curva CRF × WACC

Capital Recovery Factor em função do WACC

  • Eixo X: WACC (4% → 15%)
  • Eixo Y: CRF = r(1+r)^n / ((1+r)^n − 1)
  • Uma linha por tecnologia (vida útil diferente)
  • Marcador vertical: WACC atual = 8%
Tecnologias com vida útil longa (solar 25a) são mais sensíveis ao WACC — uma variação de 8% para 12% aumenta o CRF em ~40%.
📊 Waterfall LCOE

Composição do custo nivelado (LCOE implícito)

  • Para cada tecnologia: decomposição
  • Barra 1: parcela CAPEX × CRF (R$/MWh baseado em FC médio)
  • Barra 2: OM fixo / (8760 × FC)
  • Barra 3: CVU
  • Total: LCOE estimado em R$/MWh
O LCOE depende do FC médio de cada tecnologia/subsistema — linkando com a View 3, pode-se calcular LCOE regionalizado.
🌡 Heatmap sensibilidade

LCOE × (WACC, CAPEX)

  • Eixo X: WACC (4% → 15%)
  • Eixo Y: CAPEX ± 30% do valor PDE
  • Cor: LCOE resultante
  • Um painel por tecnologia expansível (sol, eol)
Identifica em qual combinação de WACC+CAPEX uma tecnologia passa a ser mais cara que a outra — define limiares de decisão.

Questões estratégicas que cada view responde

Por que visualizar cada dado — decisões de modelagem que dependem dessas análises
  • Onde estão as lacunas geográficas de geração renovável? View 1 (mapa SIGA) → identifica regiões sem usinas solares/eólicas existentes ou planejadas — candidatas a instalação no modelo de expansão. Relevante para decidir quais subsistemas devem ser marcados como is_expansivel = true.
  • Qual é a vida útil remanescente do parque instalado? View 1 (timeline SIGA) → usinas que entram em operação antes de 2000 podem sair do sistema antes do horizonte 2035, reduzindo a capacidade base. Hoje o modelo usa p₀ como fixo — isso seria uma subestimação do investimento necessário.
  • O perfil de demanda mensal é suficiente ou precisa de resolução semanal/diária? View 2 (série + box-plot mensal) → se o desvio padrão intra-mensal for >10% da média, a hipótese de demanda constante por mês introduz erro significativo nas horas de pico.
  • Solar e eólico se complementam dentro de um mesmo subsistema? View 3 (heatmap FC hora×mês) → se solar tem pico diurno e eólico tem pico noturno/madrugada, a combinação reduz a necessidade de armazenamento ou backup térmico.
  • Quão correlacionado é o vento entre subsistemas diferentes? View 3 (scatter correlação cruzada) → baixa correlação NE↔S valida o benefício do intercâmbio de energia eólica. Alta correlação implicaria que o modelo precisaria de maior capacidade de transmissão ou geração distribuída.
  • O parque eólico/solar real é homogêneo dentro de cada subsistema? View 3 (violin por usina raw) → se houver usinas com FC muito abaixo da média do subsistema, a agregação por subsistema pode superestimar a disponibilidade real.
  • Qual é o gargalo de transmissão inter-regional mais crítico? View 4 (matriz capacidade) → pares com baixa capacidade relativa à demanda do subsistema destino limitam o aproveitamento de excedentes renováveis de outras regiões.
  • As linhas de transmissão ativas são resilientes ou dependem de poucos circuitos? View 4 (scatter km × MVA) → um par que depende de 1–2 linhas longas de alta tensão é vulnerável — o modelo assume capacidade garantida, mas a realidade tem indisponibilidade.
  • O WACC de 8% é sensível o suficiente para inverter a ordem de competitividade sol vs. eol? View 5 (heatmap LCOE × WACC × CAPEX) → se o LCOE solar só supera o eólico acima de WACC=12%, o resultado do modelo é robusto a variações realistas de taxa.
  • O bloco_mw mínimo de expansão reflete o mercado atual de EPC? View 5 (grouped bar) → 30 MW solar e 50 MW eólico são valores razoáveis para leilões de 2024, mas projetos menores existem. Alterar esse parâmetro muda a granularidade da solução ótima.
  • A proxy de disponibilidade (N←NE, SE_CO←S) é uma simplificação aceitável? View 3 (ranking e violin por usina raw) → comparar o FC médio do subsistema proxy com o real estimado a partir das usinas SIGA naquele subsistema. Se divergir >15%, vale criar uma fonte primária para N e SE_CO.

Stack sugerida para implementação

Recomendações para um dashboard interativo que leia diretamente do SQLite

Opção A — Observable Framework

  • Lê SQLite direto via DuckDB WASM ou SQLite WASM no browser
  • Markdown + JavaScript → HTML estático deployável
  • Plot (Observablehq) para a maioria dos charts
  • Leaflet ou MapLibre para o mapa SIGA
  • Vantagem: zero backend, arquivo único, SQL direto
  • Desvantagem: o db tem 727MB — não embarca no browser; precisaria de DuckDB server ou pré-export de parquets

Opção B — Streamlit (local)

  • Python nativo — mesmo ambiente do projeto
  • SQLAlchemy session já configurada
  • Plotly para charts interativos
  • Folium/Pydeck para mapa
  • Vantagem: sem ETL extra, acesso total ao ORM, roda no laptop
  • Desvantagem: requer servidor local, não é HTTP estático puro

Opção C — Export → HTML estático

  • Script Python exporta todos os dados relevantes para JSON/parquet
  • HTML com D3.js ou Vega-Lite consome os JSONs embutidos
  • Totalmente offline, shareable via e-mail
  • Vantagem: portátil, sem dependências
  • Desvantagem: dados estáticos — não reflete novas instâncias do mart sem re-exportar
Recomendação: Dado que o projeto já tem Python/SQLAlchemy configurado, a sequência natural é Streamlit para exploração rápida (Opção B) e depois exportar as views finais como HTML estático auto-contido (Opção C) para compartilhamento. O mapa SIGA (10k usinas com lat/lon) é a view que mais ganha com interatividade real.
